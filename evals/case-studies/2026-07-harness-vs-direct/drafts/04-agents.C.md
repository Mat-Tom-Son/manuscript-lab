# Agents as Operators

Manuscript Lab treats agents as its primary distribution surface [cite:readme]. The delivery mechanism is `mlab mcp`, a zero-dependency Model Context Protocol server that runs over stdio [cite:mcp]. Any MCP client — Claude Code, Claude Desktop, Cursor, or a custom agent — connects and receives the Manuscript Lab primitives as typed tools [cite:mcp].

The server does not invent a parallel API. Each tool call executes the local `mlab` CLI against the workspace, so the file protocol remains the sole source of truth [cite:mcp]. An agent calling `gate_target` runs the same `mlab gate` command a human would type, with the same arguments, against the same files.

What the agent gets beyond raw access is structure and safety. Tools are typed: arguments are validated through a shared catalog that enforces project-relative paths, bounded integers, and allowlisted identifiers, rejecting invalid input with a JSON-RPC error before anything executes [cite:mcp]. Tool annotations advertise posture to the client — `readOnlyHint`, `destructiveHint`, and `openWorldHint` — so an agent or its harness can reason about side effects before calling [cite:mcp]. Exposure flags control which tools are visible at all. The default surface includes only reads and generated-state writes, excluding anything that calls a model or requires human approval [cite:mcp]. `--read-only` restricts further to pure project reads. `--all-tools` unlocks the full catalog, including approval-gated and model-calling tools like review, chorus, and export [cite:mcp]. Draft prose, exports, and release actions are unreachable without that explicit opt-in.

Consider a concrete flow. An agent joins a project where a long draft already exists. It calls `validate_project` to confirm the workspace is well-formed, then `status_project` to see which sections are `todo` and which word counts sit below the 33% floor that blocks release [cite:readme]. The report names each blocker and the exact command that closes it, so the agent has a work list rather than a vague signal [cite:readme]. The agent calls `adopt` against the existing draft if the workspace is not yet initialized, or proceeds directly to `check_static` on a target section to run the deterministic document checks [cite:mcp]. Static checks return typed issues — missing contract fields, unresolved claims, citation gaps — written to the issue ledger as durable work items, not ephemeral chat advice [cite:readme]. The agent revises the draft file to address each issue, re-runs `check_static`, and iterates until static checks pass. Then it calls `gate_target` against the section, which evaluates the readiness gate from accumulated evidence: contract status, word count, claim resolution, citation integrity [cite:readme]. The gate writes its verdict to state. If the verdict is green, the section is ready. If not, the gate reports which evidence requirement still fails, and the agent loops back to revision.

Nothing in this loop bypasses the file protocol. The agent operates the same harness a human operates, with the same gates deciding readiness from the same evidence spine. The MCP server adds typed access, safety annotations, and exposure control — not a shortcut around verification.
