# Agents as Operators

Manuscript Lab treats agents as the primary operators of the harness, not as secondary users who happen to share a CLI. The distribution surface for agents is `mlab mcp`, a zero-dependency Model Context Protocol server over stdio [cite:mcp]. Any MCP client — Claude Code, Claude Desktop, Cursor, or a custom agent — connects to the server and receives the Manuscript Lab primitives as typed tools: validation, status, checks, gates, reports, evidence, and artifact inspection [cite:mcp]. Each tool call executes the local `mlab` CLI against the workspace. Nothing bypasses the file protocol, and readiness still comes from checks, issues, and gates [cite:mcp].

The server exposes three safety postures through exposure flags. The default posture includes only tools that never call models and never require human approval: pure project reads plus generated-state writes under the workspace's state directory [cite:mcp]. The `--read-only` flag narrows further to tools whose effects are pure reads [cite:mcp]. The `--all-tools` flag opens the full catalog, including approval-gated and model-calling tools such as review runs, writers' room blue-sky, chorus, merge apply, export, and done [cite:mcp]. Tool annotations advertise each tool's posture to the client: `readOnlyHint` for pure reads, `destructiveHint` for writes or release actions, and `openWorldHint` for tools that call a model [cite:mcp]. Arguments are validated by the same shared catalog used by the CLI driver — project-relative paths only, no traversal, no absolute paths, bounded integers, allowlisted identifiers — and invalid arguments are rejected with JSON-RPC `-32602` before anything runs [cite:mcp].

A concrete worked flow shows how an agent operates the harness from an existing draft to a readiness verdict. The agent begins by adopting a manuscript with `mlab adopt existing-draft.md`, which copies every markdown file into contracted `draft/` sections without modifying the originals [cite:readme]. The agent then calls `status_project` to inspect the workspace state. A freshly adopted draft will report blockers: short imported sections sit below the word floor, purposes are marked TODO, and claims have no sources yet [cite:readme]. The agent calls `report_project` to get the full blocker list, where each blocker carries the exact command that closes it [cite:readme]. The agent works through the list: it calls `check_static` to run deterministic document checks and identify missing scaffolding, then revises the draft files to fill in section contracts, add sources to claims, and bring prose above the 33% word-count floor [cite:readme]. The agent calls `check_static` again to confirm the issues are resolved. It calls `citations_check` to verify citation markers resolve against registered sources, and `evidence_report` to confirm the evidence spine is intact [cite:mcp]. Finally, the agent calls `gate_target` against the section or the full manuscript. The gate engine decides readiness from evidence — `section-ready`, `citation-ready`, `manuscript-ready`, and `export-ready` — and returns a verdict [cite:readme]. If the gate passes, the agent reports the manuscript as ready. If it blocks, the gate names the failing requirement ids and the agent loops back to revision.

Throughout this flow, the agent never receives a silent rewrite or an unaccountable suggestion. Every action is a tool call against the file protocol, every issue is a durable work item in the ledger, and every readiness verdict is a gate decision backed by evidence [cite:readme]. The `AGENTS.md` file in the repo serves as the standing contract for agents working in a Manuscript Lab workspace, codifying these expectations [cite:readme].
