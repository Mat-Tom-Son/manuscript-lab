# Style

## Voice

Clear, modest, and operational. Prefer short paragraphs, concrete nouns, and
plain verbs. The prose should sound like a careful technical note, not a product
launch.

## Formatting

- Use Markdown headings no deeper than `###`.
- Keep examples runnable where practical.
- Use code formatting for file paths, commands, ids, and literal values.
- Avoid decorative language, hype, and unexplained acronyms.

## Terminology

- Use "context packet" for the composed bundle under `state/runtime/`.
- Use "issue" for a durable review finding that can be accepted, rejected,
  deferred, merged, or closed.
- Use "candidate" for an alternative full-section revision.
- Use "audit" for a before/after revision check.
- Use "export" for reader-facing Markdown or HTML output.

## Citation Rules

This fixture should avoid external technical claims. When it makes a local
fixture claim, list it in `state/claims.md` and connect it to a source key in
`sources/index.md`.
