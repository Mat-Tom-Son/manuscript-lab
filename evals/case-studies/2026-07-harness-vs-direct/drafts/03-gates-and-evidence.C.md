# Gates and Evidence

"Done" is not a feeling. In Manuscript Lab 2.0, it is a verdict produced by the gate engine, a readiness layer that turns existing checks, issue state, runtime packet state, and claim/source state into named, repeatable decisions with machine-readable evidence [cite:gate-engine]. The engine does not score literary quality. It answers a narrower question: does this target satisfy the configured readiness standard, and what evidence supports that answer?

A gate is a named readiness contract. The core gates are `section-ready`, `manuscript-ready`, `citation-ready`, and `export-ready`. Each gate contains requirements with stable IDs, severities, sensors, and evidence payloads. A requirement is a single check inside a gate. Profiles adjust which requirements are included and at what severity without changing the gate ID. The `default`, `draft`, `release`, and `ci` profiles let the same gate serve a writer's local pass and a CI pipeline's hard gate.

The `section-ready` gate illustrates the requirement model. Its blocking requirements include `contract.valid`, which checks that the section file conforms to its declared section contract; `runtime.fresh`, which verifies that the runtime packet is not stale relative to the source files; and `issues.no_blockers`, which confirms the issue ledger contains no open blockers for that target [cite:gate-engine]. A warning-level requirement, `reviews.latest_clean`, reports whether the latest review pass left errors. Each requirement produces a structured result with a status, a deterministic flag, and an evidence payload that other tools can consume without parsing prose.

The `citation-ready` gate depends on the evidence spine, the file protocol that connects draft claims to registered sources [cite:evidence-spine]. The evidence spine tracks what the manuscript asserts, where each assertion appears, and what source support justifies it. Claims live in a claims register with statuses such as `supported`, `unsupported`, `needs-review`, and `not-needed`. A `supported` claim must name a source key present in the source index, and that key alone is not sufficient evidence; the support link must include a locator specific enough for another reader to inspect [cite:evidence-spine]. The 2.0 release moved citation requirements into the shared `evidence.*` namespace so that gate results, CI annotations, and overrides all refer to the same stable IDs.

Gate requirements that fail produce issue objects with a `requirement_id`, `severity`, `message`, and `remediation` field. The `remediation` field carries the command that fixes the blocker, so a report does not just describe a problem; it names the action. For example, if `runtime.fresh` fails because the packet is stale, the issue's remediation field reads `mlab packet build draft/04-market.md` [cite:gate-engine]. A human or an agent reads the report, runs the command, and re-runs the gate. This loop replaces ad hoc fixes with a deterministic cycle: observe, remediate, re-evaluate.

Overrides are allowed but not hidden. An override is a human decision to accept a known failed requirement for a scoped target. The result artifact records the override, the actor, the reason, and the age. An overridden requirement remains visibly overridden in the persisted result. Overrides are scoped to a single target, never global.

The gate engine writes immutable run artifacts under `state/gates/runs/` and updates a latest pointer under `state/gates/latest/`. Each artifact records the config, target hashes, requirement outcomes, overrides, warnings, errors, and final readiness. This persistence means a readiness verdict is auditable after the fact. A CI job, a human reviewer, or an agent can inspect the artifact and confirm what was checked, what passed, what failed, and what was overridden. The verdict is not a feeling or a self-reported status field. It is a file on disk with a hash and a timestamp.
