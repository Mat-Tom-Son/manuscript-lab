# The Problem

A writer asks an AI assistant to tighten a chapter on pricing. The assistant returns a cleaner paragraph, but in the process drops a qualifier the writer had spent three days sourcing. The change looks like an improvement at the sentence level. The writer does not notice until a reviewer flags the now-unsupported claim weeks later. This is the first failure mode: silent rewrites. The model overwrites prose without leaving a diff the writer can inspect, compare, or reject [cite:product-strategy].

The second failure mode is lost context. A long manuscript exceeds any single context window. The assistant working on chapter seven does not know what chapter three promised, what the outline requires, or what the section's stated purpose was. It produces text that reads well in isolation but contradicts an earlier section or drifts from the brief. The writer catches some of these drifts manually. Most slip through. There is no machine-checked contract that ties a section's prose back to a declared purpose, a target length, or a set of acceptance criteria [cite:readme].

The third failure mode is unverifiable claims. A model generates a paragraph asserting that documentation teams recover forty percent of review time after adopting a particular workflow. The claim has no registered source. The writer ships the draft. A reader asks for the citation and there is none. The manuscript has no claim register, no source manifest, and no gate that blocks export when a factual claim is unsupported. The claim is not a lie; it is an artifact with no provenance and no accountability [cite:product-strategy].

These three failures share a root cause. AI-assisted writing tools treat prose as output. They do not treat it as an artifact with a contract, a revision history, an evidence chain, and a release gate. Software engineering solved this class of problem decades ago with continuous integration: every change is checked against declared standards before it merges, and a failed check produces a typed, actionable issue rather than a vague opinion. Manuscript Lab applies the same discipline to long-form prose [cite:readme].

The thesis is one sentence: Manuscript Lab is local CI for prose, giving long-form writing the section contracts, typed issues, candidate trails, evidence spine, and release gates that software takes for granted [cite:product-strategy].

The tool is not a book generator [cite:product-strategy]. It does not promise quality. It promises that a section marked ready has passed named, configured checks, that every revision left a trail, and that every blocking claim has a registered source [cite:readme]. Where the model's output is good, the protocol makes that goodness verifiable. Where the model's output is bad, the protocol makes the failure visible and fixable before it reaches a reader.
